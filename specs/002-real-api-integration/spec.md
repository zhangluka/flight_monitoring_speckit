# Feature Specification: 航班搜索数据源切换为真实接口

**Feature Branch**: `002-real-api-integration`  
**Created**: 2026-03-05  
**Status**: Draft  
**Input**: User description: "现在的数据是mock数据，如何更换为真实可用的接口数据"

## Clarifications

### Session 2026-03-05

- Q: 真实接口不可用时是否必须支持降级（mock/缓存）？ → A: 不要求降级；仅需展示用户可理解的错误信息，降级能力不在本 feature 范围内。
- Q: 配置缺失或无效时系统行为？ → A: 回退到 mock；配置缺失或无效时使用内存 mock 数据源，保证搜索仍可用（便于开发/演示）。
- Q: 真实接口返回数据与展示契约不一致或缺失时系统行为？ → A: 尽力映射；能映射的字段照常展示，缺失或无法映射的字段用占位或隐藏，记录日志，不向前端暴露原始错误。
- Q: 真实接口失败时是否要求运维可用的日志/指标？ → A: 需要；真实接口失败时必须记录可被运维使用的日志（如错误类型、请求标识），便于排查与监控，不要求额外指标系统。
- Q: 本 feature 是否要求同时支持多个真实 API 数据源？ → A: 单一即可；仅支持当前配置的一个真实 API，通过配置在真实 API 与 mock 之间切换，不要求主备或多源。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 从真实数据源获取并展示航班 (Priority: P1)

用户发起航班搜索时，系统从已配置的数据源（真实航班搜索接口）获取结果并展示，替代当前仅返回内存 mock 的行为。

**Why this priority**: 核心价值是「数据来自真实接口」，无此则需求未满足。

**Independent Test**: 在配置为真实数据源的前提下，用户提交有效出发地、目的地与日期并搜索，结果列表中的航班来自该接口且非 mock；可独立验证数据源切换与展示链路。

**Acceptance Scenarios**:

1. **Given** 已配置真实航班搜索数据源，**When** 用户提交有效的出发地、目的地与出行日期并搜索，**Then** 系统展示来自该数据源的航班列表（非 mock）。
2. **Given** 真实数据源返回多条航班，**When** 结果渲染完成，**Then** 列表展示与现有展示契约一致（航班号、起降时间、舱位与价格等）。
3. **Given** 用户修改搜索条件后再次搜索，**When** 提交新搜索，**Then** 系统向真实数据源发起新请求并展示对应结果。

---

### User Story 2 - 通过配置切换数据源 (Priority: P2)

运维或部署人员可通过配置（如环境变量或部署配置）在「真实 API」与「mock/降级」之间切换数据源，无需修改代码或重新构建。

**Why this priority**: 支持开发、演示与生产环境灵活切换，降低运维成本。

**Independent Test**: 修改数据源相关配置并重启或重新加载后，用户执行相同搜索，系统使用新数据源返回结果；可独立验证配置生效。

**Acceptance Scenarios**:

1. **Given** 当前使用 mock 数据源，**When** 运维将配置改为真实 API 并完成部署/重启，**Then** 后续用户搜索使用真实 API 返回结果。
2. **Given** 当前使用真实 API，**When** 运维将配置改为 mock 或降级源并完成部署/重启，**Then** 后续用户搜索使用 mock 或降级源返回结果。
3. **Given** 配置未指定或无效，**When** 系统启动或处理搜索，**Then** 系统回退到内存 mock 数据源，搜索仍可用；该行为在文档中说明。

---

### User Story 3 - 真实接口不可用时的错误与降级 (Priority: P3)

当真实接口不可用、超时或返回错误时，用户看到可理解的错误信息或降级结果，且不暴露内部错误堆栈或敏感细节。

**Why this priority**: 保障体验与安全，在 P1/P2 满足后的健壮性增强。

**Independent Test**: 通过模拟真实接口超时或错误响应，验证用户端展示的错误信息可理解且无内部细节泄露。

**Acceptance Scenarios**:

1. **Given** 真实接口超时或返回不可用，**When** 用户发起搜索，**Then** 用户在合理时间内（如 10 秒内）看到明确、可理解的错误提示（本 feature 不要求提供 mock/缓存降级）。
2. **Given** 真实接口返回 4xx/5xx 或业务错误，**When** 系统处理响应，**Then** 用户看到的文案可理解且不包含堆栈、内部路径或密钥信息。

---

### Edge Cases

- 无可用数据源配置或配置为空时，系统回退到内存 mock 数据源，保证搜索可用；该行为在文档中明确。
- 真实接口返回的数据与现有展示契约（航班号、起降时间、舱位、价格等）不一致或缺失时，系统尽力映射：能映射的字段照常展示，缺失或无法映射的字段用占位或隐藏；记录日志便于排查，不向前端暴露原始错误。
- 真实接口限流或配额超限时，用户应看到可理解的提示（如「请求过于频繁，请稍后再试」），且凭证与配额信息不暴露给前端。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系统必须支持从当前配置的单一「真实航班搜索接口」获取数据，并替换当前仅使用内存 mock 的行为；不要求同时支持多个真实 API 或主备切换。
- **FR-002**: 数据源选择必须由配置（如环境变量或部署配置）决定，不写死在代码中；配置缺失或无效时，系统必须回退到内存 mock 数据源。
- **FR-003**: 真实接口的请求参数必须包含出发地、目的地、日期，并与现有搜索入口一致；返回数据在服务端尽力映射为现有展示所需的航班列表（航班号、起降时间、舱位与价格等），缺失或无法映射的字段用占位或隐藏并记录日志。
- **FR-004**: 系统必须将接口错误、超时或不可用转化为用户可理解的错误信息；本 feature 不要求实现 mock 或缓存降级。
- **FR-005**: 凭证与 API 密钥仅保存在服务端配置中，不暴露给前端。
- **FR-006**: 真实接口调用失败（超时、4xx/5xx、网络错误）时，系统必须记录可被运维使用的日志（至少包含错误类型，可选请求标识），便于排查与监控；不要求额外指标或追踪系统。

### Key Entities _(include if feature involves data)_

- **数据源配置**: 标识使用真实 API 还是 mock；仅支持一个真实 API 实例，其端点与认证方式由配置提供，不写死具体供应商名称。
- **航班搜索结果**: 与现有一致（航班号、起降时间、出发地/目的地、舱位与价格）；真实接口响应需在服务端映射为该结构后再返回前端。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 用户在使用「真实数据源」配置时，搜索得到的航班列表来自该接口，且单次搜索响应在 5 秒内完成（或符合所选接口的 SLA）。
- **SC-002**: 通过更改配置即可在「真实 API」与「mock/降级」之间切换，无需代码变更或重新构建。
- **SC-003**: 当真实接口不可用时，用户能在 10 秒内看到明确错误提示，且无未捕获异常或内部错误堆栈暴露给用户。

## Assumptions

- 真实接口的选型、注册与合规已由项目 research（如 001-flight-search 的 research）覆盖；本规格不指定具体供应商或 API 名称。
- 现有前端与 BFF 的请求/响应契约保持不变，仅 BFF 内部数据获取由 mock 改为可配置的真实接口调用。
